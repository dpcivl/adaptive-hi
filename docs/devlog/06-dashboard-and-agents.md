# 06 · Day 4 — 발표 대시보드 + 역할분리 에이전트 + 비교 강화

3백엔드 비교를 "더 단단하게" 만들고(질문 보강·토큰 차등), 발표용 Streamlit
데모를 붙이고, 작업 워크플로를 전담 에이전트로 분리한 단계.

## 구현

| 파일 | 역할 |
|---|---|
| `app.py` | Streamlit 발표 데모 — 트랙별 데이터 차트 + 질문→3백엔드 답변·메트릭 나란히 비교 + 검색 출처/원문 |
| `evals/questions.py` | 함정·추론형 질문 4개 추가(총 2→6). 오탐 유도·코퍼스밖·압축기 진단·피크 저감 |
| `config.py` | `MAX_TOKENS_BY_TRACK`(synthetic 1024 / steel 2048) + `DEFAULT_MAX_TOKENS`, env 조정 |
| `src/analysis/pipeline.py` | `answer(max_tokens=None)` → 미지정 시 트랙별 기본값 조회 |
| `.claude/agents/*.md` | 역할분리 에이전트 4종(code-writer / code-reviewer / answer-grader / rag-auditor) |

## 설계 결정

- **질문 4개는 모두 코퍼스+데이터 근거**. 질문 속 kW 수치(367/49/56/150/338/320)를
  brief가 산출하는 실제 값과 일치시켜, 모델이 보는 입력과 어긋나지 않게 했다.
  - 함정 2종: 정상 설비 오탐 유도(생산라인 주간 피크), 코퍼스밖 수치 유도(ESS 회수기간 → '자료에 없음' 기대).
  - 추론 2종: 압축기 주말 고착 진단(언로더/누설), 피크 저감(기본요금=15분 피크 개념).
  - 함정형은 **로컬 vs API의 환각 차이**를, 추론형은 **추론력 차이**를 드러내는 게 목적.
- **max_tokens 트랙 차등**: 512→1024로 올렸으나 steel(개방형 관리자 질문)은 1024에서도
  절단됨 → steel 2048, synthetic 1024. `answer()`가 트랙으로 해석, app 슬라이더는 트랙 기본값에서 시작.
- **역할분리 에이전트**: 작성(writer)과 검증(reviewer)을 분리하되, **reviewer에 Write/Edit를
  주지 않아** 리뷰어가 직접 고치지 못하게 강제 — 문제는 writer에게 반려. 채점(grader)·검색검수(auditor)는
  read-only 판정자. reviewer 제1원칙은 "주석을 믿지 말고 git diff로 실제 동작 확인".

## 적대적 리뷰

- 확인: max_tokens 해석이 synthetic→1024/steel→2048/unknown→1024(방어적), 슬라이더 경계
  `max(2048, default)`로 env override 시도 깨짐 방지.
- 확인: `run_analysis`는 코드 변경 없이 트랙 차등 자동 적용(인자 없이 호출).
- 확인: 새 변경에 비밀키·주입·외부입력 없음. config는 기존 env-override 스타일과 동일.
- 한계: steel 2048도 "절대 안 잘림" 보장은 아님(관측 절단점의 2배 헤드룸). 비용은 steel 답변이
  길어져 소폭 증가(의도된 것, 답변당 ≈$0.05 수준).

## 채점 실험 (answer-grader 루브릭 시범)

steel 즉석 질문("연간 패턴→관리자 절감안")을 5축(정답성/환각/출처/언어/유용성)으로 채점.
검색 청크와 인용을 한 줄씩 대조해 **사실 검증**:

- **Claude 9/10**: M&V 산정식·시각화 인용이 검색 청크 원문과 일치(환각 0), 주말 패턴을 옳게
  재해석. 단 **1024에서 절단** → 이번 트랙 차등의 직접 근거가 됨.
- **gpt-4o 7/10**: 간결·빠름(4.8s/$0.01)이나 가장 풍부한 M&V 가이드라인을 안 쓰고 "커패시터"
  등 자료 밖 일반지식 주입(프롬프트 "지어내지 말 것" 위반 소지).
- **exaone 7/10**: 수치는 맞히되 데이터 산 값을 PDF에 오귀속, 자기 메타주석으로 산만.

> 교훈: "근거 충실·정확한 출처 귀속" 품질에서 API가 값을 했고(답변당 ≈$0.04), 그 차이를
> **LLM-as-judge로 정량화**하는 것 자체가 이 프로젝트의 talking point.

## 워크플로 변화

이번 세션부터 기능 작업은 4개 에이전트로 분담(`.claude/agents/`, 다음 세션부터 호출 가능):
writer(작성) → reviewer(적대적 리뷰+커밋) → grader(답변 채점) → auditor(RAG 검수).
