# 03 · 데이터 품질 게이트 (readable-ratio)

실 PDF를 코퍼스에 넣은 뒤, 저품질 청크가 인덱스를 오염시키지 않도록 청킹 단계에
품질 게이트를 추가했다.

## 배경: 코퍼스 검증에서 발견

`scripts/inspect_corpus.py`로 351 청크를 측정 → 무결성·크기는 정상이나 **7개 청크가
readable_ratio < 0.6**. 실제 내용을 확인하니 M&V 가이드라인의 **목차(점선 dot-leader)
페이지**였다. 손상은 아니지만 RAG 가치가 낮다(질의가 본문 대신 목차에 매칭될 위험).
→ "측정하지 않았으면 못 잡았을" 문제. 검증의 가치.

## 구현

| 파일 | 내용 |
|---|---|
| `src/rag/documents.py` | `readable_ratio()` (한글+ASCII 비율, 싼 휴리스틱) + `load_documents()`가 임계 미만 청크 제외, 문서별 제외 로깅 |
| `config.py` | `MIN_READABLE_RATIO=0.5` (env 튜닝 가능) |
| `scripts/inspect_corpus.py` | 재현 가능한 품질 검사 도구 (문서별 청크 수·크기 분포·품질·무결성) |

## 설계 결정과 대안 거절

- **청크 단위 필터** (문서 전체 제외 아님): M&V는 94%가 정상, 목차만 문제.
- **임계 config화**: 코퍼스마다 다를 수 있어 하드코딩 회피.
- **langdetect/LLM 품질검사 거절**: 관측된 실패(점선 목차)엔 과설계 + 의존성.

## 적대적 리뷰와 판단 (맹종하지 않음)

리뷰 결과 blocker/major 없음. 엣지케이스(0으로 나눔, chunk_id 연속성, 전체 제외
문서, 순환 import) 전부 정상 확인.

MINOR 1건: `·℃—` 등 정상 문자를 "나쁨"으로 분류 → 밀집 청크 오탐 위험.
**리뷰어의 화이트리스트 제안은 거절**했다 — `·`가 목차 점선의 본체라, 화이트리스트에
넣으면 목차도 통과되어 게이트가 무력화된다. 대신 **임계를 0.6→0.5로 낮춰 오탐 마진을
확대**(목차 ≤0.25는 여전히 제외, 우리 코퍼스 영향 0).

## 테스트 결과

- `dropped 7/116` (M&V), 총 351 → **344**, 무결성 344 == 344.
- min readable_ratio **0.08 → 0.91**, below-0.6 청크 **0**.
- 검색 정상 + 목차 대신 **실제 M&V 본문**(공기압축기 15분 kWh 데이터 수집)이 검색됨.

## 한계 (정직성)

숫자·표 위주 고-ASCII 청크나 한글 코드포인트 mojibake는 이 게이트로 못 거른다.
관측된 유형(점선 목차·font garble)만 대상. 더 어려운 케이스가 나오면 신호(언어감지
등)를 추가하는 것이 다음 단계.
